## Project 1: Execution Witness
- [developeruche] Preliminary exploration of new method `engine_newPayloadWithWitness`
    - Highlight: [code](https://github.com/developeruche/witness-bench), [report](https://hackmd.io/@0xdeveloperuche/S1YYKLDwWx). This endeavor was pursued to practically determine if introducing this new method would be efficient. Three clients were explored during this experiment: Geth, Ethrex, and Reth (using ExEx witness indexing). The Geth coupled flow (`engine_newPayloadWithWitness`) provided a 1.5x–3.4x speedup, eliminating historical rewind penalties. However, this didn't appear to be the case in Ethrex—although both its flows were significantly faster than Geth and Reth, its decoupled flow actually offered better performance than its coupled one. The report also explores changing the transport layer, yielding interesting results.
    - Raw Custom TCP protocol for witness retrieval: [spec](https://hackmd.io/@0xdeveloperuche/SkScRpTObe), [rust-lib](https://github.com/developeruche/witnet). The major bottlenecks identified in the benchmark report were the transport protocol and JSON encoding. These were addressed by sending raw binary data over a dedicated TCP stream instead of JSON-RPC. This improved performance drastically: throughput increased from ~500 Mbps to ~12 Gbps, and the time to ship a 500MB witness dropped from 11 seconds to ~200 milliseconds.
    - Hot take: It is safe to say that, based on the data presented above, it would be highly beneficial to introduce a new method `engine_newPayloadWithWitness` to the Engine API that transports the witness over a dedicated TCP stream instead of JSON-RPC. Ideally, upgrading the transport layer used by the Engine API would yield similar performance improvements to this experiment.

## Project 6: Benchmarking and Metrics

- [Ignacio] EEST Benchmarks now targeting Fusaka has been run for 10M, 30M and 60M gas in a 8x5090s prover. Additionally, the dashboard UI has been improved to show more details about the benchmark results. You can see the dashboard [here](https://eth-act.github.io/zkevm-benchmark-runs/repricing/). Note that this is still not an optimized prover from the hardware perspective, thus the measurements might not reflect still the optimal setup for the Zisk zkVM.
- [Ignacio] A new [Proof Verification dashboard](https://eth-act.github.io/zkevm-benchmark-runs/verification/) is release and long-term maintained. This dashboard shows measurements of proof verification times for multiples zkVMs for machines with [targeted hardware setup](https://eips.ethereum.org/EIPS/eip-7870). Additionally, proof sizes and security bits extracted from [soundcalc](https://github.com/ethereum/soundcalc) are also included as extra information.
- [Ignacio] Reth and Ethrex guest programs have been benchmarked using Zisk `ziskemu` emulator to measure their "AIR surface cost". This provides a hardware agnostic measurement of the cost of execution the guest programs for mainnet blocks. More information can be found in the [report](https://hackmd.io/@jsign/eth-zisk-air-cost-reth-ethrex).
- [Rafael] [Benchmarkoor](https://github.com/ethpandaops/benchmarkoor) now supports configuring custom EL RPC calls when running benchmarks. The motivator for this feature was allowing collecting `debug_executionWitness` outputs when benchmarks are run in state-bloated chains (i.e. bloatnet) e.g. SLOAD/SSTORE/BALANCE related cases.

### M3 — Figure out what the end-game for specs looks like
 - [Cody] Finish and begin to circulate ZisK proving system [draft executable spec](https://github.com/codygunton/pil2-proofman). The idea is that if enough key stakeholders get on board, this could become the official spec which could be reinforced by strong testing, rather than diverging immediately, as formal written specs often do. A guide (which will bit rot) to help readers navigate the Python spec is [here](https://codygunton.github.io/pil2-proofman/). Testing: 
   - For "simple" AIRs, the Python prover produces exactly the same proofs as the GPU prover (i.e., byte-for-byte equivalent).
   - For RISC-V AIRs, the Python verifier verifies GPU prover proofs, with some failure tests.
   - The Python verifier also can verify VADCOP proofs, which are recursive aggregations.
   - There is also a Lean4 verifier with similar testing.

### **M6:** Identify the minimum acceptable formal verification requirement:
  - [Cody] A stunning paper was published this month on the subject of formal verification: [_Verification Theatre: False Assurance in Formally Verified Cryptographic Libraries_](https://eprint.iacr.org/2026/192) by Nadim Kobeissi. This makes it clear that we should place tight, extremely explicit, and simple descriptions of the verification boundary and the claimed guarantees provided by formal proofs.
