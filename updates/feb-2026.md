## Project 1: Execution Witness
- [developeruche] Preliminary exploration of new method `engine_newPayloadWithWitness`
    - Highlight: [code](https://github.com/developeruche/witness-bench), [report](https://hackmd.io/@0xdeveloperuche/S1YYKLDwWx). This endeavor was pursued to practically determine if introducing this new method would be efficient. Three clients were explored during this experiment: Geth, Ethrex, and Reth (using ExEx witness indexing). The Geth coupled flow (`engine_newPayloadWithWitness`) provided a 1.5x–3.4x speedup, eliminating historical rewind penalties. However, this didn't appear to be the case in Ethrex—although both its flows were significantly faster than Geth and Reth, its decoupled flow actually offered better performance than its coupled one. The report also explores changing the transport layer, yielding interesting results.
    - Raw Custom TCP protocol for witness retrieval: [spec](https://hackmd.io/@0xdeveloperuche/SkScRpTObe), [rust-lib](https://github.com/developeruche/witnet). The major bottlenecks identified in the benchmark report were the transport protocol and JSON encoding. These were addressed by sending raw binary data over a dedicated TCP stream instead of JSON-RPC. This improved performance drastically: throughput increased from ~500 Mbps to ~12 Gbps, and the time to ship a 500MB witness dropped from 11 seconds to ~200 milliseconds.
    - Hot take: It is safe to say that, based on the data presented above, it would be highly beneficial to introduce a new method `engine_newPayloadWithWitness` to the Engine API that transports the witness over a dedicated TCP stream instead of JSON-RPC. Ideally, upgrading the transport layer used by the Engine API would yield similar performance improvements to this experiment.
